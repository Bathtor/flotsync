# Flotsync OKF Documentation Conventions

## Purpose

Flotsync uses OKF-style metadata to make design documents easier to discover,
cross-link, and check. OKF metadata organises the documentation corpus; it does
not replace prescriptive protocol specs, Rust APIs, protobuf schemas, tests, or
`bd` issue tracking.

## Concept Documents

Concept documents live under `docs/` and use Markdown with YAML frontmatter.
`docs/index.md` is generated and intentionally has no frontmatter.

Required frontmatter fields:

- `type`: one known concept type.
- `title`: human-readable concept title.
- `description`: one concise sentence used in generated indexes.
- `status`: one known status value.

Optional frontmatter fields:

- `applies_to`: repo paths, crate names, or package names affected by the doc.
- `source_of_truth`: paths that are more authoritative than the concept doc.
- `implements`: concept docs this document implements or refines.
- `depends_on`: concept docs this document relies on.
- `supersedes`: concept docs this document replaces.
- `tracked_by`: `bd` issue ids for active design or implementation work.

Known concept types:

- `API Contract`
- `Benchmark`
- `Crate Overview`
- `Decision`
- `Protocol`
- `Scenario`
- `State Machine`
- `Testing Guide`
- `Wire Format`

Known status values:

- `draft`: still being worked through or known to be incomplete.
- `settled`: intended direction is agreed, but implementation coverage may vary.
- `implemented`: current code or checked-in artefacts implement the document.
- `superseded`: kept for history, but replaced by another concept.
- `deferred`: intentionally postponed.

## Body Conventions

Keep the existing document body unless the user asks for a design rewrite. When
adding structure to a new document, prefer the established local pattern:

- start with scope and non-goals where relevant
- define terminology before rules
- document ownership boundaries explicitly
- keep invariants and compatibility expectations checkable
- keep open or deferred questions in the document, not as Markdown tasks

Do not add OKF `log.md` files. Git and `bd` already handle history and work
tracking.

## Link Rules

Use relative Markdown links inside `docs/`. Prefer linking to the concept doc
that owns a topic rather than to an incidental mention. Keep code and proto links
relative to the document being edited.

Relationship fields such as `depends_on`, `implements`, and `supersedes` should
refer to existing concept documents by `docs/`-relative path when used, for
example `communication_protocol_spec.md`.

## Generated Index

`docs/index.md` is generated by:

```bash
amm scripts/okf-docs.sc generate-index
```

Do not hand-edit generated index content. To change the index, edit concept
frontmatter and regenerate it.

Validate the corpus with:

```bash
amm scripts/okf-docs.sc check
```

The checker should reject missing required fields, unknown concept types or
status values, duplicate titles, broken local links, relationship fields that
point nowhere, and stale generated index content.
